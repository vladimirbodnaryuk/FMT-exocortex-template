# Жёсткие различения (Hard Distinctions)

> **ЗАПРЕЩЕНО путать эти пары.** Критические онтологические ошибки.
> Источник: FPF (A.7 Strict Distinction)

## 1. Система ≠ эпистема

| ❌ НЕ путать | ✅ Правильно | Тест |
|-------------|-------------|------|
| Система = область знания | **Система** ≠ эпистема | Можно ли указать на границу в пространстве? |
| "Система мышления" | Системное мышление — это метод, не система | Есть ли у "системы" 4D-границы? |
| "Система управления" = практика | Различай: орг.система vs практика управления | Есть ли владелец, входы/выходы? |

**Частый случай:** «система знаний» — ❌. Знания не являются системой (нет 4D-границ). Корректно: «экзокортекс» (конкретная система) или «набор описаний» (содержимое репозиториев).

**Частый случай:** «система типов» — ❌. Типы — это классификация, не физический объект. Корректно: «типизация», «четыре типа», «классификация репозиториев».

## 2. Метод ≠ Инструмент

| ❌ | ✅ |
|---|---|
| "Я делаю Toggl" | "Я веду учёт времени в Toggl" |
| "Перешёл с учёта на Clockify" | Сменил инструмент, не метод |

**Тест:** Можно ли описать метод без названия инструмента?

## 3. Рабочий продукт ≠ описание

| ❌ | ✅ |
|---|---|
| "У меня есть стратегия" (без документа) | Нарратив ≠ продукт |
| "Моя архитектура минималистичная" | Слова ≠ артефакт |

**Тест:** Можно ли передать это другому человеку без слов?

## 4. Учёт ≠ Планирование

| Учёт (accounting) | Планирование (planning) |
|---|---|
| Что было (past) | Что должно быть (future) |
| Дескриптивно | Прескриптивно |

**Тест:** Это запись факта или намерения?

## 5. Роль ≠ Агент; Агент ≠ Инструмент; Агентская роль ≠ Функциональная роль (DP.D.023, DP.D.033)

| ❌ | ✅ |
|---|---|
| "Стратег — отдельный агент" | Стратег — **роль** (R1). Claude — holder (исполнитель) |
| "Бот — агент" | Бот — инструмент (I1, Grade 1), играет роль Проводника (R13) |
| "Синхронизатор — агент" | «Синхронизатор» — и роль (R8), и инструмент (I2). Разные ракурсы |
| "Роль требует агентности" | **Агентская** роль (A.13) требует Grade 2+. **Функциональная** (A.2) — нет |

**Role-Centric (DP.D.033):** Роль описывается независимо от исполнителя. Исполнитель (holder) выбирается и подготавливается отдельно. Одно имя = и роль, и система (как «молоток»).
**Нотация FPF:** Holder#Role:Context@Window
**Каталог ролей:** DP.ROLE.001 §3.2 (R1-R20). Реестр исполнителей: §3.1 (A1-A2, I1-I8).
**Подготовка исполнителя:** program (скрипт) | prompt (LLM) | train (человек).
**Тест 1:** Может ли сущность выбирать, КАК действовать? Да → агент. Нет → инструмент.
**Тест 2:** Это описание функции (что делать) или описание системы (чем делать)? Функция → роль. Система → исполнитель.

| Метод | Навык |

**Тест:** Это то, что записано, или то, что заключено?

## 9. Артефакт ≠ Процесс

| Артефакт | Процесс |
|---|---|
| Продукт (существительное) | Деятельность (глагол) |

**Тест:** Можно ли указать на это после завершения?

## 10. Pack-контент ≠ Governance-контент

| Pack (source-of-truth) | Governance (управление) |
|---|---|
| **Что истинно** в предметной области | **Что делаем** в проекте |
| Знания, архитектуры, методы | Планы, сроки, реестры |
| Универсально (другой проект) | Локально (этот проект) |

**Тест 1:** Можно ли использовать в другом проекте? Да → Pack. Нет → Governance.
**Тест 2:** "Как устроено?" → Pack. "Что делаем?" → Governance.

## 11. Обещание (= сценарий использования) ≠ Описание метода ≠ Сервис (DP.D.039)

**Три понятия + навигатор:**

| Обещание / Сценарий использования (SC) | Описание метода (MethodDescription) | Сервис (accessSpec) |
|---|---|---|
| ЗАЧЕМ и ЧТО получит потребитель | КАК работает (шаги) | ГДЕ точка входа (навигация) |
| U.ServiceClause | U.MethodDescription | accessSpec в MAP.002 |
| `Pack DP / 08-use-cases/` | `<repo>/PROCESSES.md` | `MAP.002` (каталог) |
| Не зависит от реализации | Зависит от системы | Зависит от архитектуры |

**Описание метода имеет два масштаба** (не два разных понятия):

| Внутрисистемное («процесс») | Межсистемное («сценарий») |
|---|---|
| Шаги внутри одной системы | Управление передаётся между системами |

**Синонимы:**
- «Сценарий использования» = «обещание» = SC (U.ServiceClause). Можно использовать взаимозаменяемо.
- «Сценарий» (без «использования») = межсистемное описание метода. НЕ синоним обещания.
- «Процесс» = внутрисистемное описание метода.

**Метафора:** Обещание = меню ресторана («вы получите стейк»). Сервис = дверь на кухню. Описание метода = рецепт повара (процесс — на одной кухне, сценарий — маршрут от склада через кухню до стола).

**Тесты:**
- Описывает, ЧТО получит потребитель (без КАК)? → Обещание (SC).
- Описывает шаги реализации? → Описание метода (процесс или сценарий).
  - Внутри одной системы? → Процесс.
  - Управление передаётся другой системе? → Сценарий.
- Это точка входа для навигации (кнопка/команда)? → Сервис.

## 12. Описание ≠ Знание

| Описание (артефакт) | Знание (capability) |
|---|---|
| В репозитории (FPF, SPF, Pack) | В агенте (человек, ИИ) |
| Копируемо | Требует обучения |
| Проверяется чтением | Проверяется выполнением задачи |

**Тест:** Можно ли передать файлом? Да → описание. Нет → знание.
Подробно: `PACK-personal/.../01B-distinctions.md` D.013, `PD.FORM.004`

| Мировоззрение (в агенте) | Описание (в репо) |
|---|---|
| Определяет КАК читать | Содержит ЧТО написано |
| «Компилятор» | «Исходник» |

**Тест:** FPF — описание. То, как агент интерпретирует FPF — мировоззрение.
Подробно: `PACK-personal/.../01B-distinctions.md` D.015

## 14. Стратегический DDD ≠ Тактический DDD (DP.D.017)

| Стратегический (SOTA) | Тактический (ООП-карго-культ) |
|---|---|
| BC, Context Map, UL | Entity, VO, Aggregate, Repository |
| Универсален (любой стек) | ООП-специфика |

**Тест:** Работает без ООП-языка? Да → стратегический. Нет → тактический.

## 15. Model ≠ View (DP.D.018)

Model = изменение семантики (Pack). View = изменение формы (Downstream).

**Тест:** Нужно обновить Pack? → model. Достаточно downstream? → view.

## 16. DSL ≠ DSLM (DP.D.019)

DSL = явный язык (SQL, Terraform). DSLM = LLM fine-tuned на домене.

**Тест:** Формальный парсер? → DSL. Обученная модель? → DSLM.

## 17. Capture During-Work ≠ After-Work (DP.D.020)

During = в момент обнаружения (SOTA). After = ретроспективно (потери).

**Тест:** В той же сессии? → during-work. Позже? → after-work (потери неизбежны).

## 18. Уровни принципов: 0-й ≠ 1-й ≠ 2-й ≠ 3-й

| Level | Что | Репо-тип | Фреймворк |
|-------|-----|----------|-----------|
| 0 (нулевые) | Транс-дисциплинарные мета-ограничения | Foundation (ZP) | Нет |
| 1 (первые) | Фундаментальные основания дисциплин | Framework (FPF) | = бандл с принципами |
| 2 (вторые) | Принципы предметных областей | Pack | SPF (отдельно) |
| 3 (третьи) | Ситуационные, проектные | Downstream | S2R и другие (несколько) |

**Тест:** Работает в любой области? → 0-й. В любой дисциплине? → 1-й. В конкретном домене? → 2-й. В конкретном проекте? → 3-й.

> Pack = вторые принципы. Downstream = третьи принципы. Подробно: `ZP/README.md`

## 21. Обязанность → Одна роль → Один сценарий (Anti-Bystander)

| ❌ | ✅ |
|---|---|
| «3 агента чистят WP» | «R1 Стратег, сценарий Session-Prep — единственный владелец архивации WP» |
| Обязанность дублируется в нескольких сценариях | Обязанность принадлежит одной роли, одному сценарию |
| Все должны → никто не делает (bystander effect) | Один владелец → всегда выполняется |

**Правило:** Одна обязанность = одна роль = один сценарий. Другие сценарии могут **помечать** (status: done), но **не выполнять** основное действие.

**Тест:** Если убрать один из N сценариев, выполняющих обязанность — сломается ли что-то? Если нет → дублирование (bystander). Если да → зависимость (ок).

**Пример исправления (21 фев 2026):** Архивация done WP-файлов была в Close (шаг 7), Note-Review (шаг 8), Session-Prep (шаг 8.5). Все три пропускали. Решение: Close и Note-Review только помечают status → Session-Prep перемещает.

## 22. «Уровень» = строго часть-целое (физическое вхождение)

| ❌ НЕ путать | ✅ Правильно |
|-------------|-------------|
| «4 уровня системы» (Ecosystem→Platform→Template→Personal) | «4 контура системы» — организационные контуры, не физическое часть-целое |
| «На каком уровне?» (про контур ответственности) | «В каком контуре?» — контур = граница ответственности и governance |
| «Уровень» = масштаб, зона, контур | **Уровень** = строго U.SystemLevel: уровень 2 физически входит в уровень 1 (как двигатель в автомобиль) |

**Правило:** Слово «уровень» допустимо ТОЛЬКО для физического часть-целого (автомобиль → двигатель → поршень). Для организационных границ, зон ответственности, масштабов → «контур», «зона», «масштаб». НИКОГДА «уровень».

**Тест:** Уровень 2 полностью физически входит в уровень 1? Да → уровень. Нет → контур (или другое слово).

## 23. Platform ≠ IWE Template ≠ Personal IWE (4 контура системы)

| Контур | Что | Governance | Кто меняет |
|--------|-----|-----------|-----------|
| **L2: Platform** | Инфраструктура, сервисы (бот, MCP, DB) | Pack DP, MAP.002 | Разработчик платформы (M2) |
| **L3: IWE Template** | Шаблон для пользователя (CLAUDE.md + memory/ + стратег) | FMT-exocortex-template | Разработчик шаблона (M3) |
| **L4: Personal IWE** | Экземпляр пользователя (настроенный, подключённый) | Личный CLAUDE.md | Пользователь IWE (M4) |

**Тесты:**
- Это затрагивает всех пользователей? → L2 (Platform). Только шаблон? → L3 (Template). Только меня? → L4 (Personal).
- WakaTime: L3 рекомендует (опционально), L4 настраивает.
- Новый MCP-сервер: L2 (Platform) — нужен Pack DP entity + MAP.002.

**IntegrationGate:** Перед интеграцией нового инструмента/агента → определи контур + тип + роли + продукты + процессы. Нет ответов → НЕ устанавливать.

## 24. Проект ≠ Система

| Проект (деятельность) | Система (вещь) |
|---|---|
| Временна́я: начало и конец | 4D-границы, существует после завершения проекта |
| Создаёт/изменяет систему | Выполняет функцию |
| «Строительный проект» | «Здание» |

**Правило:** «Твой проект — это система» — ❌. Проект создаёт систему, но сам системой не является. Корректно: «Здание, которое ты строишь — это целевая система. Проект — деятельность по его созданию.»

**Тест:** Можно ли указать на это в пространстве после завершения проекта? Да → система. Нет → проект (деятельность).

## 25. Черновик ≠ Заготовка (стадии творческого конвейера)

| Черновик (private draft) | Заготовка (public draft) |
|---|---|
| Личный, никому не показывается | Опубликована, видна другим |
| Может быть сырым, незаконченным | Связный текст, понятный читателю |
| Цель: кристаллизация мысли для себя | Цель: тестирование гипотез, обратная связь |

**Тест:** Можно ли показать этот текст другому человеку без доработки? Да → заготовка. Нет → черновик.
Подробно: PD.FORM.005 §3

## 26. Заготовка ≠ Пост (стадии творческого конвейера)

| Заготовка (draft) | Пост (published) |
|---|---|
| status: draft — видна, но не продвигается | status: published — активно продвигается |
| Может быть переработана, разделена, объединена | Версия зафиксирована, ревизия допустима |
| TTL ≤14 дней | Без TTL |

**Тест:** Готов ли автор активно привлекать внимание к этому тексту? Да → пост. Нет → заготовка.
Подробно: PD.FORM.005 §3

## 27. Бот ≠ Платформа; Neon + DT MCP = один ЦД (DP.ARCH.003)

| ❌ НЕ путать | ✅ Правильно |
|-------------|-------------|
| Neon DB = база бота | Neon = **платформенная БД ЦД** (L2 Platform) |
| DT MCP = отдельная система от Neon | Neon + DT MCP = **слои одного ЦД** |
| Бот — основной источник событий ЦД | **Основные источники:** LMS и Клуб. Бот — дополнительный (новички, онбординг) |
| Бот владеет схемой development | Бот — **клиент**: вызывает `log_event()`, предполагает что таблица существует |

**ЦД = одна система из нескольких хранилищ:**
- **DT MCP** (Cloudflare) — декларативный слой (Layer 0): профиль, цели, контекст
- **Neon `public.users`** — Identity Layer: telegram_id ↔ ory_id ↔ dt_user_id
- **Neon `development.user_events`** — Layer 1: лог событий (append-only)
- **Neon `development.engagement`** — Layer 2: метрики (SQL View)
- Будущее: skill_mastery, memory_decay, qualifications (Layer 2 проекции)

**Операционное состояние бота ≠ данные ЦД:**
- `development.user_state` содержит и ЦД-данные (прогресс, streak), и чисто операционные (current_state FSM, current_context). Операционное состояние — собственность бота, не часть ЦД.

**Продьюсеры событий (по убыванию значимости):**
1. LMS (Aisystant) — основной: ДЗ, курсы, тесты
2. Клуб (Discourse) — основной: посты, комментарии, peer-review
3. Бот — дополнительный: онбординг, марафон для новичков
4. Web App (будущий) — прямая запись


**Тест 1:** «Кто владеет этой таблицей?» → платформа (L2) vs бот (L3 instrument). Если платформа → миграция вне бота.
**Тест 2:** «Это данные О человеке или операционное состояние системы?» О человеке → ЦД. Состояние системы → бот.

## 28. Протез ≠ Экзоскелет (паттерн взаимодействия ИИ с человеком)

| Протез (prosthesis) | Экзоскелет (exoskeleton) |
|---|---|
| ИИ заменяет функцию мышления | ИИ усиливает функцию мышления |
| Вход: «Что мне делать?» → Выход: готовый ответ | Вход: «Вот что я думаю» → Выход: дополнение |
| Человек принимает/отклоняет | Человек думает сам, ИИ усиливает |
| Результат: задача решена, человек не научился | Результат: задача решена + человек стал компетентнее |
| Атрофия: навык видеть аффордансы деградирует | Рост: навык видеть аффордансы тренируется |

**Три механизма экзоскелета:**
1. **Предъявление, а не генерация** — ИИ показывает знание пользователя в нужный момент, не генерирует своё
2. **Вопросы, а не ответы** — в стратегических решениях ИИ спрашивает «а ты как думаешь?» до выдачи ответа
3. **Убывающие подмостки (scaffolding)** — помощь уменьшается с ростом компетенции

**Дизайн-принцип IWE (DP.ARCH.001 #21):** IWE — экзоскелет, не протез. Критерий: после взаимодействия пользователь стал компетентнее.

**Тест:** Кто после взаимодействия стал умнее — пользователь или только ИИ получил больше данных?

## 32. Три класса задач по стоимости проверки (Левенчук 2026)

| Класс | Проверка | AI-автономия | Пример |
|-------|----------|-------------|--------|
| **Closed-loop** (тесты) | Дешёвая, автоматическая | Высокая (commoditized) | Код с unit-тестами, математика, криптография |
| **Open-loop** (инженерия) | Дорогая, отложенная | Средняя (люди+агенты) | Марсоход, продукт, реальный SE |
| **Problem framing** | Неизвестная (эволюция) | Низкая (люди ведут) | Стратегия, «развитие для развитых» |

**Следствие для IWE:** Платформа работает на классах 2 и 3. Класс 1 — commoditized, не фокус.
**Следствие для координации:** Класс 2 требует процессного фреймворка (ОРЗ, WP, captures). Класс 3 требует творчества с проверками (семинары, экзокортекс).
**Закон Брандолини:** Чушь предложить легко, опровергнуть — дорого. В классах 2-3 верификация ≥ генерации по стоимости.

**Тест:** Сколько стоит проверить решение по сравнению с его генерацией? Дёшево → класс 1. Дорого → класс 2. Неизвестно → класс 3.

## 36. Экзокортекс ≠ IWE

| Экзокортекс | IWE |
|---|---|
| Подсистема: память и инструкции (CLAUDE.md + memory/) | Среда: все 5 архитектурных видов (DP.IWE.001 §3) |
| 2 вида: Системы (файлы) + Описания (содержимое) | 5 видов: + Роли + Методы + Рабочие продукты |
| componentOf IWE | Целое |

**Тест:** Затрагивает только файлы инструкций/памяти? → Экзокортекс. Затрагивает роли, методы или рабочие продукты? → IWE.

**Маркетинговый синоним:** «Экзокортекс 2.0» допустим в публичных текстах как синоним IWE. В доменном языке платформы — не использовать.

## 37. BYOB ≠ Managed (модель данных пользователя, DP.D.036)

| BYOB (Bring Your Own Backend) | Managed (Multi-tenant) |
|---|---|
| L4-данные на ресурсах **пользователя** (его Neon/Supabase/SQLite) | L4-данные на ресурсах **платформы** (общий Neon, tenant_id) |
| Платформа даёт только код + L2-знания | Платформа хранит всё |
| GDPR: пользователь отвечает за свои данные | GDPR: платформа = data processor |
| Trust = 0 (данные не уходят) | Trust > 0 (доверие к платформе) |

**Решение IWE:** BYOB (ADR-018 v2). Knowledge Gateway (локальный) объединяет L2 (remote, платформа) + L4 (BYOB, пользователь) → единый endpoint.

**Тест:** Данные пользователя покидают его машину/аккаунт? Да → Managed. Нет → BYOB.

## 38. Верификация ≠ Валидация (VR.D.001)

| Верификация | Валидация |
|---|---|
| «Сделано ли **правильно**?» (соответствие эталону) | «Сделано ли **нужное**?» (пригодность для цели) |
| Эталон: спецификация, Pack-сущность, принципы | Эталон: потребность пользователя, цель |
| Может автоматизировать агент | Чаще требует человека |

**FM:** Подмена верификации валидацией — артефакт «работает» (валиден), но нарушает принципы (не верифицирован).

## 39. Аудит ≠ Ревью (VR.D.002)

| Аудит | Ревью |
|---|---|
| Систематический обход: **полнота** (coverage) | Экспертная оценка: **качество** (depth) |
| Роль: R24 Аудитор | Роль: R25 Рецензент |
| Автоматизируем (Haiku/Sonnet) | Требует экспертизы (Opus/человек) |

**Правило:** Сначала аудит (всё ли есть?), потом ревью (правильно ли?).

## 40. Эталон ≠ Шаблон (VR.D.003)

| Эталон (reference) | Шаблон (template) |
|---|---|
| Содержание + форма: «как должно быть» | Только форма: «какая структура» |
| Для верификации (сравнить) | Для создания (заполнить) |
| Pack-сущность, принцип | SPF pack-template |

**FM:** Верификация по шаблону = только формальная (VT.001). Для содержательной (VT.002) нужен эталон.
