## § Масштаб: День (Day Open)

> **Триггер:** «открывай» / «открывай день» / «открой день»
> **Роль:** R1 Стратег
> **Формат:** Claude Code показывает в чат → в конце записывает в DayPlan (git)

### Алгоритм Day Open

#### 1. Вчера

Собрать коммиты за вчера по всем репо:

```bash
for repo in $(ls ~/IWE/); do
  if [ -d ~/IWE/$repo/.git ]; then
    commits=$(git -C ~/IWE/$repo log --since="yesterday 00:00" --until="today 00:00" --oneline --no-merges 2>/dev/null)
    [ -n "$commits" ] && echo "=== $repo ===" && echo "$commits"
  fi
done
```

Сопоставить с DayPlan вчера → вывести 1-3 ключевых результата.

#### 2. План на сегодня

- Прочитать текущий `WeekPlan W{N}...` из `DS-my-strategy/current/`
- Выбрать 2-4 РП на сегодня
- **Правило слота:** Если есть РП с тегом саморазвитие / учёба / курс / руководство → **всегда первый** в таблице
- **Если саморазвития нет** → явно сказать: *«Сегодня нет саморазвития в плане. Добавить?»*

#### 3. Саморазвитие

- Показать текущее руководство (из `DS-principles-curriculum/` или контекст из WeekPlan)
- Где остановился (из WP context file)
- Список активных черновиков: `ls DS-my-strategy/drafts/` (≤5 = норма)

#### 4. Стратегирование

- Проверить: сегодня день стратегии? (конфиг пользователя, по умолчанию = понедельник)
- Если да → *«Сегодня — день стратегии. Запланируй сессию стратегирования.»*

#### 5. IWE за ночь

- Проверить логи автоматики (launchd, cron-скрипты):
  - sync-agent, note-review, reindex — отработали?
  - Ошибки?
- Новые коммиты от автоматики (git log --since="yesterday 18:00")

#### 6. Мир

- Дайджест из RSS: AI, инженерия, политика (топ-3 новости)
- Источник: RSS-агрегатор (если настроен) или WebSearch

#### 7. Запись

- Обновить или создать `DS-my-strategy/current/DayPlan YYYY-MM-DD.md`
- Если предыдущий DayPlan → `git mv → archive/day-plans/`
- Закоммитить в DS-my-strategy

## § Масштаб: Сессия (Session Open)

> **Триггер:** Любое задание от пользователя (кроме триггеров Day Open/Close).
> **Роль:** R6 Кодировщик

## WP Gate (блокирующее)

**ОБЯЗАТЕЛЬНО. ДО ЛЮБОГО ДЕЙСТВИЯ по задаче.**

1. Прочитать MEMORY.md → секция «РП текущей недели»
2. Совпадает с РП? → Да: п. 3. Нет: СТОП.

**ЗАПРЕЩЕНО до проверки:** читать код, исследовать, планировать, вызывать агентов.

### Совпадает — работаем

1. Ссылаемся на номер РП.
2. **DayPlan Gate:** Проверить `DS-my-strategy/current/DayPlan YYYY-MM-DD.md`. Если РП **нет в DayPlan** текущего дня → добавить строку в таблицу (бюджет, приоритет, статус). Это гарантирует, что дневной план отражает фактическую работу.
3. Переходим к Ритуалу.

### Не совпадает — СТОП

1. «Этой задачи нет в плане на неделю.»
2. Вывести таблицу РП из MEMORY.md
3. Спросить: артефакт, формулировка РП, репо, бюджет
4. Предложить перестановку РП если бюджет ограничен
5. Дождаться подтверждения
6. Записать **в четыре места** (атомарно):
   - `MEMORY.md` → таблица РП (новая строка)
   - `DS-my-strategy/docs/WP-REGISTRY.md` → новая строка в таблицу (сортировка: от последнего к первому)
   - `DS-my-strategy/current/WeekPlan W{N}...` → таблица РП (новая строка)
   - `DS-my-strategy/inbox/WP-{N}-{slug}.md` → context file (всегда: описание, источник, контекст, критерий готовности)
   > **Нумерация:** `{N}` — только последовательные целые числа (74, 75…). Буквенные суффиксы (73a, 73b) запрещены — каждый РП самостоятелен.
7. Только после записи во все три — начать

### Исключения (РП не нужен)

- Задачи ≤15 мин
- Вопросы/исследования без изменений файлов
- Экстренные баг-фиксы (пользователь говорит «срочно»)

**Эскалация:** Если сессия начата по исключению, но пользователь задаёт второе задание → *«Это перерастает в РП. Записать?»*

> **Урок WP Gate (2026-02-09):** Переход от исследования к действию = переход через WP Gate. Даже если исследование было исключением, момент «теперь исправь» требует полной проверки. Никогда не квалифицировать задачу как исключение самостоятельно, если результат — файлы.

## Ритуал согласования

> После WP Gate, перед работой.

**Шаг 1.** Определить роль пользователя и объявить:
> *«**Роль пользователя:** [предложение из 4 ролей — memory/roles.md]. **Роль Claude:** [из каталога DP.ROLE.001 §3.2]. **Работа:** [что]. **РП:** [артефакт]. **Метод:** [как]. **Оценка:** [~Xh]. **Модель:** [текущая] — рекомендую [модель] ([причина]).»*

> **Роль пользователя:** Предложить, из какой из 4 ролей Tseren даёт задание (HD #31): (1) Разработчик платформы, (2) Пользователь платформы, (3) Разработчик своего IWE, (4) Пользователь своего IWE. НЕ угадывать молча — всегда предложить и дождаться подтверждения.
> Роли Claude нет в каталоге → *«Роль не найдена. Предлагаю добавить: [название].»*

**Рекомендация модели:**
- Opus — архитектура, сложный код, multi-system изменения, стратегия
- Sonnet — типовые задачи, одно-файловые правки, написание контента
- Haiku — быстрые поиски, простые вопросы, тривиальные фиксы
- Пользователь может не согласиться и поменять модель перед началом работы

**Шаг 2.** Дождаться согласования (включая модель).

**Шаг 3.** Определить файлы/репо, ограничения. Если есть context file (`DS-my-strategy/inbox/WP-{N}*.md`) — прочитать.

> **Критичное чтение:** Context file может устареть. Иерархия доверия: (1) реальные файлы/код → (2) свежие документы → (3) WP context / WORKPLAN.

**Исключения:** Задачи §2.4 освобождены от WP Gate, но Ритуал (объявление) — всегда. Продолжение работы над тем же РП — повторный Ритуал не нужен.

**Шаг 4. Регистрация сессии** (после согласования):
> Добавить строку в `DS-my-strategy/inbox/open-sessions.log`:
> ```
> YYYY-MM-DD HH:MM | WP-N | модель | краткое описание задачи
> ```
> Если файл не существует — создать. Исключения (≤15 мин) — не регистрировать.

## Зонтичные РП (Umbrella WP)

Для систем с непрерывной работой — два постоянных РП:
- **#N-dev** (развитие) — фичи из MAPSTRATEGIC
- **#N-debt** (техдолг) — баги, latency, cleanup

Context file = живой backlog (10-20 пунктов). Done удаляются. Каждую неделю — конкретный scope в WeekPlan.

**Отсечка:** задача ≥4h или отдельный lifecycle → эпизодический РП.

## Issue Funnel (Воронка → WP-debt) — v2 DB-native

**Двухуровневый триаж (R7 Триажёр):**
- **Auto-triage (Grade 1):** helpful=false / comment → Haiku classify → `feedback_triage` DB → TG alert if severity>=high
- **Review (Grade 3):** при открытии сессии техдолга → structured report + 2 файловых intake

**Источники:** `feedback_triage` DB, `unsatisfied-questions.md` (REPORT, не inbox), `fleeting-notes.md`, `captures.md`.

**Review-протокол (при открытии сессии техдолга):**
1. Прочитать unsatisfied-questions.md
2. Прочитать fleeting-notes + captures
3. Прочитать WP-debt backlog
4. Review + оценить бюджет
5. Одобренные → backlog, отработанные → `status='resolved'`

**Прямая команда:** «запиши замечание: X» → fleeting-notes.md с тегом бот.
